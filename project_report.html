Phoenix SmartCaseAI - Application Report

Phoenix SmartCaseAI is a full-stack QA acceleration platform that converts Jira issues or manual requirements into structured test cases, generates locator strategies and automation templates, supports review workflows, exports test assets, and pushes validated cases to TestRail.

Scope: Problem Context, Objectives, Features, Detailed Functionalities, Architecture, API Surface, Data Flow, Security/Reliability, Tech Stack, Benefits, Limitations, and Future Roadmap

Executive Summary

Phoenix SmartCaseAI is designed to accelerate software QA operations by reducing the manual cycle required to convert requirements into test artifacts. The system combines requirement ingestion (Jira/manual), AI-assisted generation, human review controls, export utilities, and TestRail publishing into one workflow. It enables QA teams to produce repeatable outputs while keeping reviewers in control of final quality.

Problem Statement

  • QA teams spend significant time manually translating requirement text into structured test cases.
  • Test case quality and style can vary between authors, affecting consistency across releases.
  • Artifact handoff to tools like TestRail often requires repetitive formatting and data entry.
  • Locator generation and automation test bootstrap are frequently delayed due to manual DOM analysis.

Project Objectives

Primary

  • Generate structured and practical test cases from Jira or manual inputs.
  • Support quick reviewer edits and enforce export/push guardrails.
  • Enable one-click export in multiple QA-friendly formats.
  • Push approved cases to TestRail with configurable repository strategy.

Secondary

  • Provide locator intelligence and runnable automation templates from raw DOM.
  • Support multi-provider LLM routing with fallback behavior for reliability.
  • Allow attachment-aware generation for richer test coverage.
  • Keep sensitive credentials strictly backend-side.

Core Features

1) Test Case Generation

  • Generate from Jira issue keys (single or combined issue references).
  • Generate from manual input (description, acceptance criteria, custom prompt, attachments).
  • Supports test types such as functional, regression, UI, security, performance, create, update, edge.
  • Assigns normalized test case IDs and enforces consistent output structure.

2) Locator Generation

  • Accepts DOM/HTML as input.
  • Produces locator JSON with primary and alternate locator strategies.
  • Outputs runnable automation templates for Selenium/Playwright in Python/Java/TypeScript.
  • Includes model attribution so teams know which provider generated the output.

3) Review + Publishing

  • Inline editing of generated test cases with edit metadata tracking.
  • Review status model: approved / pending / rejected.
  • Rejected cases are excluded from export and TestRail push operations.
  • Supports duplicate-case hints and change-diff visualization in edit mode.

Business Benefits

  • Faster QA planning: Reduces manual effort for test design and formatting.
  • Higher coverage: Multi-type generation improves scenario breadth.
  • Better consistency: Standard schema and export formats align QA deliverables.
  • Toolchain integration: Direct push to TestRail supports operational QA workflows.
  • Flexible AI strategy: Supports multiple providers plus fallback generation for resilience.

Functionalities in Detail

Input Handling

  • Jira issue parsing: summary, description, acceptance criteria, custom fields, attachments.
  • Manual uploads: PDF, DOCX, TXT, CSV, XLSX, XML, JSON, YAML, HTML, and more.
  • Attachment processing with parse status and download status visibility.
  • Attachment open/download proxy with host-level allow checks for safer retrieval.

Generation Engine

  • Prompt-builder with strict JSON schema instructions.
  • RAG-style semantic context retrieval using sentence embeddings + FAISS.
  • Provider routing for Gemini, OpenAI, Anthropic, OpenRouter, local Llama, fallback rules.
  • Response parser normalizes and recovers JSON fragments when model output is noisy.

Outputs

  • Test case payloads with IDs, steps, priorities, and expected results.
  • Exports: Excel, PDF, Gherkin, plain text, CSV, JSON.
  • TestRail push modes: single repository, baseline mode, multiple test suites.
  • Review pipeline persists current test case snapshot before export/publish actions.

Architecture & Workflow

User Input (Jira/Manual/DOM) -> React UI -> Flask API -> Jira/Parser/LLM Engine -> Response Parser -> Review Table -> Export or TestRail Push
  • /generate-from-jira and /manual-generate-test produce test cases.

Operational Flow (Detailed)

  1. User selects workflow tab and submits input payload with model preference.
  2. Frontend validates required fields and invokes corresponding backend API.
  3. Backend resolves source content (Jira/manual + attachments).
  4. LLM engine enriches context (embedding search) and builds strict JSON prompt.
  5. Provider call is executed; response parser normalizes returned JSON into schema.
  6. Generated cases are shown in review table; reviewer can edit/compare changes.
  7. Reviewed snapshot is posted to backend; rejected cases are excluded downstream.
  8. User exports artifacts or pushes approved cases to TestRail.

Security, Reliability, and Quality Controls

  • Credentials remain backend-side via environment variables.
  • Attachment proxy validates URL host and scheme before retrieval.
  • Input guards for required fields, allowed test types, file size, framework/language constraints.
  • LLM fallback path ensures output generation continuity during provider outages.
  • Review-stage gating prevents unsaved edits from being exported/pushed.
  • Export/TestRail only process non-rejected cases to reduce operational noise.

Current Limitations

  • Latest reviewed cases are stored in memory and may be reset on backend restart.
  • No dedicated persistent database layer for audit/history in current architecture.
  • Attachment parsing quality depends on source file format and extractable text quality.
  • LLM output quality and coverage vary by selected model/provider and prompt context quality.

Suggested Roadmap

Near-term

  • Add persistent storage for generation/review history and run metadata.
  • Add explicit approve/reject controls at row level in UI if not already enabled.
  • Add automated regression tests for API contracts and parser behavior.

Mid-term

  • Introduce role-based access, audit logs, and enterprise auth integration.
  • Improve semantic retrieval isolation per request to avoid context bleed.
  • Add workflow analytics: generation quality, review effort, and TestRail conversion metrics.
